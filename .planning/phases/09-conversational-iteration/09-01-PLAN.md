# Plan: 09-01 Change Detection & Parameter Updates

## Objective

Enable natural language design modifications that map to parameter changes and trigger re-generation.

## Context

The revise flow already exists: `reviseDesign()` injects prior parameters into context and Claude extracts new parameters. But the UX is rough — no indication of what changed, no parameter diff, and cost approval resets unnecessarily on minor tweaks. This plan polishes the iteration loop.

## Tasks

- [x] **Task 1: Parameter diff display**
      After `parameters_extracted` event on a revision (version > 1), compute and display a diff of changed parameters. Create `ParameterDiff.tsx` component showing old→new values for changed fields only. Render inline in chat after the assistant message. Backend: add `previous_parameters` to the `parameters_extracted` event payload by fetching the prior design version.

- [x] **Task 2: Smart cost re-estimation**
      Skip cost approval on parameter-only changes (no new conversation tokens needed for generation). In `usePreview`, if the design is a revision and only parameters changed (same category, same conversation), auto-approve cost. Only show cost card for first generation or category changes. Update conversation service to mark revisions with `is_revision: true` in the event.

- [x] **Task 3: Revision-aware preview**
      When a revised design generates, show a brief "Updated" indicator on the preview panel. Pass version number to PreviewPanel for display. Add subtle transition animation when STL swaps (opacity fade). Show "v2", "v3" etc. badge on preview header.

## Notes

- Revision flow already works via `reviseDesign()` in useConversation
- Design.version auto-increments, parent_design_id tracks lineage
- Keep it simple — enhance existing flow, don't rebuild
