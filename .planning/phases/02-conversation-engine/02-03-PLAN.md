  <action>
Create backend/src/app/prompts/ package.

design_wizard.py contains the system prompt as a Python string constant DESIGN_WIZARD_SYSTEM_PROMPT.

The prompt should instruct Claude to:

1. ROLE: You are a friendly 3D printing design assistant. You help people create custom 3D-printable objects by understanding what they need and asking smart clarifying questions.

2. FLOW:
   - Start by understanding WHAT they want (category detection: bracket, enclosure, organizer)
   - Then WHY (use case, context — this informs dimensions and features)
   - Then WHERE (physical context — what it mounts to, what it holds)
   - Then specifics (dimensions, features, constraints)
   - Use the request_clarification tool when you need specific info and can offer helpful options
   - Use extract_design_parameters tool ONLY when you have enough info for a complete, printable design

3. PERSONALITY:
   - Conversational, not interrogative — weave questions naturally
   - Suggest reasonable defaults ("Most phone holders are about 75mm wide — does that work for yours?")
   - Explain WHY you're asking ("I need the exact width so it fits snugly without rattling")
   - Keep it under 10 turns total — be efficient but thorough

4. DIMENSION GUIDANCE:
   - Always work in millimeters internally
   - Accept user input in any unit and convert
   - Suggest standard sizes when the user is unsure
   - For mounting: ask about screw size (M3, M4, M5), wall type
   - For enclosures: ask about what goes inside (dimensions + clearance)
   - For organizers: ask about items being stored (pens, tools, cables, etc.)

5. TOOL USE:
   - Call request_clarification when you need a specific answer and can provide helpful options
   - Call extract_design_parameters when you have ALL required dimensions and key features
   - Never guess critical dimensions — always confirm with user
   - Include sensible defaults for optional parameters

Also create a helper function get_system_prompt(conversation_context: dict | None = None) -> str that can inject additional context (e.g., from photo upload in future phases) into the base prompt.

Keep the prompt focused and under 1500 tokens — longer prompts waste context and don't improve quality.

<task type="auto">
  <name>Task 2: Build conversation orchestrator service</name>
  <files>backend/src/app/services/__init__.py, backend/src/app/services/conversation.py</files>
  <action>
Create backend/src/app/services/ package.

conversation.py: ConversationService class that orchestrates the full conversation flow.

Constructor takes: db session (AsyncSession), claude client.

Methods:

start_conversation(session_id: UUID) -> Conversation:
  - Create new Conversation record in DB
  - Return conversation object

send_message(conversation_id: UUID, user_content: str) -> AsyncGenerator[dict, None]:
  An async generator that:
  1. Load conversation messages from DB
  2. Save the new user message to DB
  3. Build messages list for Claude API (convert DB messages to Anthropic format)
  4. Call Claude with system prompt, messages, and tools (using stream_message from claude_client)
  5. Yield streamed text chunks as {"type": "text", "content": "..."}
  6. After stream completes, check for tool use in the response:
     - If extract_design_parameters: validate params with Pydantic model, save Design to DB, yield {"type": "parameters_extracted", "parameters": {...}, "design_id": "..."}
     - If request_clarification: yield {"type": "clarification", "question": "...", "options": [...]}
  7. Save assistant message (full text + any tool use) to DB
  8. On tool use, automatically send tool_result back to Claude and stream the follow-up response (Claude typically explains what it extracted after tool use)

continue_conversation(conversation_id: UUID, user_content: str) -> AsyncGenerator:
  - Same as send_message but for continuing existing conversations
  - Alias for send_message (they're the same — message history provides context)

revise_design(conversation_id: UUID, revision_request: str) -> AsyncGenerator:
  - Load existing conversation + latest design parameters
  - Inject a system message noting this is a revision: "The user wants to modify their existing design. Current parameters: {json}. Apply their requested changes."
  - Call send_message with the revision request
  - New Design record created with version incremented

get_conversation(conversation_id: UUID) -> dict:
  - Return conversation with all messages and latest design

list_conversations(session_id: UUID) -> list[dict]:
  - Return all conversations for session, ordered by updated_at desc

Handle errors: If Claude API fails, yield {"type": "error", "message": "..."} and do NOT save partial state. If Pydantic validation fails on extracted params, ask Claude to retry with the validation error.
