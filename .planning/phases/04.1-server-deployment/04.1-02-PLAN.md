<task type="auto">
  <name>Task 1: Create server provisioning script</name>
  <files>scripts/provision-server.sh</files>
  <action>
Create a bash script that provisions a fresh Ubuntu server for Docker deployment. The script should be idempotent (safe to run multiple times).

Script contents:

1. **System update:** apt update && apt upgrade -y
2. **Firewall (ufw):** Allow SSH (22), HTTP (80), HTTPS (443). Enable ufw. Deny all other inbound.
3. **Docker install:** Install Docker Engine + Docker Compose plugin via official Docker apt repo (not snap). Add current user to docker group.
4. **App directory:** Create /opt/3dez/ with proper permissions
5. **Fail2ban:** Install and enable for SSH brute-force protection
6. **Swap:** Create 2GB swap file if none exists (small VPS often need this for Docker builds)

Script should:

- Start with `#!/usr/bin/env bash` and `set -euo pipefail`
- Print what it's doing at each step
- Check if each component already installed before installing
- Be designed to run as root via `ssh root@ez3d.salundo.com 'bash -s' < scripts/provision-server.sh`

Do NOT include SSH key hardening (user already has key access). Do NOT disable password auth (user manages that).
</action>

<task type="auto">
  <name>Task 2: Create GitHub Actions deploy workflow</name>
  <files>.github/workflows/deploy.yml</files>
  <action>
Create a GitHub Actions workflow that deploys to the Hetzner server on push to main (after CI passes).

Workflow structure:

- **Trigger:** push to main branch
- **Job: deploy** — runs after CI workflow succeeds (use `needs` or separate workflow with workflow_run trigger)
  - Actually, simplest approach: single deploy.yml triggered on push to main, with a dependency on CI passing. Use `workflow_run` trigger on ci.yml completion, or just include lint/test steps. Best approach: trigger on push to main, run deploy steps directly (CI already runs on push to main separately).

**Deploy steps:**

1. Checkout code
2. SSH into server using secrets (SSH_HOST, SSH_USER, SSH_KEY, SSH_PORT)
3. Sync project files to server via rsync (exclude node_modules, .next, **pycache**, .git, .planning, .env.production)
4. SSH to run: `cd /opt/3dez && docker compose -f docker-compose.prod.yml up -d --build`
5. SSH to verify: `curl -sf http://localhost:8000/api/health` (healthcheck via backend internal port)

**Required GitHub secrets** (document in workflow comments):

- `SSH_HOST`: ez3d.salundo.com
- `SSH_USER`: root (or deploy user)
- `SSH_KEY`: Private SSH key for server access
- `ANTHROPIC_API_KEY`: For .env.production on server

**Deploy script on server** — The workflow should also write .env.production from secrets:

```bash
echo "ANTHROPIC_API_KEY=${{ secrets.ANTHROPIC_API_KEY }}" > /opt/3dez/.env.production
echo "DATABASE_URL=sqlite+aiosqlite:///./data/3dez.db" >> /opt/3dez/.env.production
```

Use `appleboy/ssh-action@v1` for SSH commands and `burnett01/rsync-deployments@7.0.1` for file sync (well-maintained Actions). Or use raw SSH with `webfactory/ssh-agent@v0.9.0` — pick whichever is simpler. Prefer the raw approach: set up SSH key, rsync files, then ssh to run docker compose.
</action>
